================================================================
  GIT BRANCH STRATEGY
================================================================

Use GitHub Flow with a develop integration branch:

  main          ← protected, production-ready
    └── develop ← integration branch (all PRs merge here)
          ├── feature/auth-refresh-token         (Member 1)
          ├── feature/doctor-availability        (Member 2)
          ├── feature/patient-medical-history    (Member 3)
          ├── feature/appointment-reminders      (Member 4)
          ├── feature/admin-analytics            (Member 5)
          └── feature/ai-report-summary          (Member 6)

Branch Naming Convention:
  feature/<member-area>-<short-description>
  fix/<member-area>-<short-description>


================================================================
  RECOMMENDED WORKFLOW
================================================================

Every member follows this workflow:

  # 1. Clone the team repo (once)
  git clone https://github.com/team-org/MediCare-Smart-Hospital.git
  cd MediCare-Smart-Hospital

  # 2. Create your feature branch from develop
  git checkout develop
  git pull origin develop
  git checkout -b feature/auth-refresh-token

  # 3. Make changes, then commit
  git add backend/controllers/authController.js
  git commit -m "feat(auth): add refresh token rotation with 7-day expiry"

  # 4. Push your branch
  git push origin feature/auth-refresh-token

  # 5. Open a Pull Request → develop on GitHub
  # 6. Another team member reviews and approves the PR
  # 7. Merge into develop (Squash or Merge Commit)
  # 8. Team leader periodically merges develop → main for releases


================================================================
  CONFLICT PREVENTION RULES
================================================================

  1. One member per controller    — No two members edit the same controller
  2. Shared utils are owner-only  — Only the assigned member edits a util file;
                                    others open a PR to the owner
  3. server.js is team leader only — Only the team leader adds new route mounts
  4. package.json needs discussion — Announce new packages in group chat first
  5. Model changes need PR review  — Schema changes affect everyone;
                                     always require at least 1 reviewer
  6. Frontend pages are isolated   — Each member's pages are in their own
                                     subfolder or route
